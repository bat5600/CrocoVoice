# Story 2.5: Créer la landing page « Créer un compte »

## Status
Proposed

## Story
**As a** CrocoVoice prospect,
**I want** une landing page dédiée au signup qui explique la valeur du compte et propose un bouton vers Supabase,
**so that** je sache pourquoi m’inscrire, je voie le lien crédible, et la future app peut simplement renvoyer vers cette page.

## Acceptance Criteria
1. Il existe un document HTML statique (`docs/signup.html` ou équivalent) qui raconte la promesse de CrocoVoice, liste les bénéfices premium, et contient un bouton “Créer un compte” qui ouvre (ou redirige vers) la page Supabase d’inscription gérée côté backend ou marketing.
2. La landing page est stylisée de façon cohérente avec le reste de l’expérience (ton clair, CTA visible, mention des fonctionnalités sync/history/paywall) et inclut les éléments de réassurance (mention paywall, sécurité des données, session Supabase).
3. Le login de l’app (story 2.4) renvoie vers cette landing lorsqu’un utilisateur clique sur “Créer un compte”, avec ouverture dans le navigateur par défaut.
4. Un “dev note” décrit où héberger la page (dans `docs/` pour tests locaux) et comment mettre à jour le lien si l’URL finale change.

## Tasks / Subtasks
- [ ] Créer `docs/signup.html` (ou `assets/signup.html`) avec sections : accroche, bénéfices premium, témoignage/bulle, CTA, footer mentionnant Supabase + paywall.
- [ ] Ajouter un court paragraphe “Pourquoi s’inscrire” incluant sync historique/dictionnaire + mention anonymisation/sécurité.
- [ ] Valider que le login ouvre cette page (mettre à jour `renderer.js` ou le flux Supabase) et documenter la logique.
- [ ] Documenter dans la story comment mettre à jour le lien vers Supabase si l’on déploie la landing ailleurs.
- [ ] Tests manuels : ouvrir l’URL, vérifier CTA, lien Supabase fonctionnel, design cohérent.

## Dev Notes
- La landing est auto-hébergée dans le repo (`docs/` ou `assets/`) pour faciliter les tests avant que le marketing fournisse une URL définitive.
- Le bouton “Créer un compte” peut être temporairement un lien vers la console Supabase (ex : `https://app.supabase.com/project/.../signup`) ou vers la future API marketing.
- L’app Electron doit ouvrir cette page dans le navigateur (via `shell.openExternal`) pour éviter de gérer le signup dans le renderer.
- Ajouter un rappel dans la landing et la doc sur la manière de modifier `config/auth.json` (ou la variable d’environnement `AUTH_SIGNUP_URL`) afin que `auth.signupUrl` reste centralisée même si la landing bouge.
- Cette story prépare la future intégration paywall; aucun backend supplémentaire n’est nécessaire pour l’instant.

### Testing
- Tests manuels uniquement.
- Scénarios :
 1. Charger `docs/signup.html` depuis un navigateur local pour vérifier l’affichage.
 2. Cliquer sur CTA → ouverture de la page Supabase/URL externe.
 3. Vérifier que l’URL est paramétrée à un seul endroit facilement modifiable si notre marketing change de domaine.
